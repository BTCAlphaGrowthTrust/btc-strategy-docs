# Access & Authentication

**All historical data is free ‚Äî no key, no sign-up.** The full catalogue, every strategy's
`stats` / `returns` / `curve`, the aligned `export` matrix, and book-level `summary` are open so you
can verify every track record against your own model *before* you commit to anything. Aliases only ‚Äî
the strategy **mechanism stays sealed**.

**The live signals are the product.** Real-time, actionable trade signals are served by a separate,
**keyed** API ‚Äî that's what you subscribe for.

## Historical data ‚Äî open, no key

| Endpoint | Access |
|---|---|
| `GET /v1/strategies` (catalogue + headline stats) | ‚úÖ free |
| `GET /v1/strategies/{id}` (metadata, provenance) | ‚úÖ free |
| `GET /v1/correlation` (cross-strategy matrix) | ‚úÖ free |
| `GET /v1/summary` (incl. `equal_weight_book`) | ‚úÖ free |
| `GET /v1/strategies/{id}/stats` ¬∑ `/returns` ¬∑ `/curve` | ‚úÖ free |
| `GET /v1/export` (aligned returns matrix) | ‚úÖ free |

Just call them ‚Äî no header required:

```bash
curl https://btc-strategy-data-api.fly.dev/v1/strategies/helios/stats
curl "https://btc-strategy-data-api.fly.dev/v1/export?period=daily"
```

That's the whole point: pull every series, drop it into your optimiser, reconcile it against your
own book ‚Äî for free. The [evaluate-against-your-book recipe](/docs/recipes/evaluate-vs-your-book)
walks the full flow.

## Live signals ‚Äî the subscription (keyed)

The **live signals API** is the product. It delivers actionable signals as your strategies fire
(`mode: "LIVE"`, `is_test: false`), gated by a per-client **`X-API-Key`** scoped to what you
subscribe to. Send the key in the **`X-API-Key` header** ‚Äî server-side only, never in a URL or a
browser bundle:

```bash
curl https://btc-signal-distribution.fly.dev/v1/signals/latest \
  -H "X-API-Key: btca_live_xxxxxxxxxxxxxxxxxxxxxxxx"
```

Live signals are **request-based** today ‚Äî [contact for access](/docs/help/contact). (There's a
commercial/compliance gate before signals are sold; no public pricing yet.)

:::warning Leverage ‚Äî signals imply leveraged positions
Following a signal at its `base_risk_pct` implies a **leveraged** position, **not a 1:1 deployment**.
Each strategy publishes **`leverage_typical`** (~1√ó) and **`leverage_max`** (historically **up to ~15√ó**
on tight-stop trades) in its metadata; each live signal carries **`implied_leverage`** for that specific
trade (`base_risk_pct √∑ stop-distance`). You execute on your own venue at your own size ‚Äî **size to your
book and cap leverage at your discretion.** Backtested / modelled, provisional ‚Äî not advice.
:::

## Response codes

| Code | Meaning |
|---|---|
| `200` | Historical data ‚Äî always, no key. |
| [`401 unauthorized`](/docs/concepts/error-codes) | **Signals API** only: missing or invalid/revoked key. Historical endpoints never need a key. |
| [`429 rate_limited`](/docs/concepts/rate-limits) | Reserved; not enforced yet. |

Every error uses the unified [`{ error, meta }`](/docs/concepts/envelope) envelope.

## CORS

Browser access is restricted to an explicit **allowlist** (never `*`):

```text
https://btcalpha.com.au
https://www.btcalpha.com.au
https://docs.btcalpha.com.au          # this docs site (so the reference playground works)
http://localhost:5173, :3000, :8100   # local development
```

Requests from an allowed `Origin` receive the matching `Access-Control-Allow-Origin`; others receive
none. Server-to-server calls (no `Origin`) are unaffected. The methods are `GET` only.

## Using the reference playground

Because `docs.btcalpha.com.au` is allowlisted, the **"Send" button** in the
[API Reference](/docs/api-reference) calls **production** directly ‚Äî and since all historical data is
open, every endpoint works with **no key**.
