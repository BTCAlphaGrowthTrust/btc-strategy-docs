# Access & Authentication

The API has **two access levels**: an open **preview** (no key) and **full** access (per-client
`X-API-Key`). The preview is the public marketing surface; full access returns deep, per-strategy
and book-level data scoped to **what your plan entitles**.

## Sending your key

Send the key in the **`X-API-Key` request header** â€” **never** in the query string or URL
(URLs leak into logs, history and referrers). One header on every request:

```bash
curl https://btc-strategy-data-api.fly.dev/v1/strategies/helios/stats \
  -H "X-API-Key: btca_live_xxxxxxxxxxxxxxxxxxxxxxxx"
```

```ts
const res = await fetch(
  "https://btc-strategy-data-api.fly.dev/v1/export?period=daily",
  { headers: { "X-API-Key": process.env.STRATEGY_API_KEY! } },  // server-side only
);
```

Use the key **server-side only** (a backend, a build step, a serverless function). Never ship it
to a browser or commit it to source control.

## Entitlements

Your key returns **only what you bought**, by tier:

- **Single** â€” one strategy's full data.
- **Bundle** â€” a curated set (e.g. the Diversifier / Flagship / Steady bundles).
- **Full catalogue** â€” every strategy **plus** book-level aggregates (`export`, full `summary`).

Requesting a strategy your plan does not include returns [`403 forbidden`](/docs/concepts/error-codes)
with a clear message â€” the catalogue itself stays visible (via the preview) so you can see what to add.

## Response codes

| Code | Meaning |
|---|---|
| [`401 unauthorized`](/docs/concepts/error-codes) | Missing **or invalid/revoked** key on a keyed request. |
| [`403 forbidden`](/docs/concepts/error-codes) | Valid key, but the strategy or feature is **not in your plan**. |
| [`429 rate_limited`](/docs/concepts/rate-limits) | Over your plan's rate limit *(reserved; not enforced yet)*. |

Every error uses the unified [`{ error, meta }`](/docs/concepts/envelope) envelope.

## Key handling

- Treat the key like a password â€” **secret, server-side, never in URLs or client bundles**.
- **Rotation:** request a new key; both work during a short overlap, then the old one is revoked.
- **Compromise:** tell us and we revoke immediately (a revoked key returns `401`).
- Caching: keyed responses are returned `Cache-Control: private, no-store` â€” do not share them across clients.

Need a key? [Request access](https://btcalpha.com.au/access).

## CORS

Browser access is restricted to an explicit **allowlist** (never `*`):

```text
https://btcalpha.com.au
https://www.btcalpha.com.au
https://docs.btcalpha.com.au          # this docs site (so the reference playground works)
https://btc-strategy-docs-k5ju.vercel.app
http://localhost:5173, :3000, :8100   # local development
```

Requests from an allowed `Origin` receive the matching `Access-Control-Allow-Origin`; others
receive none. Server-to-server calls (no `Origin`) are unaffected. The methods are `GET` only.

## Using the reference playground

Because `docs.btcalpha.com.au` is allowlisted, the **"Send" button** in the
[API Reference](/docs/api-reference) calls **production** directly. **Preview** endpoints work with
no key; **keyed** endpoints need your `X-API-Key` supplied in the playground's auth field.
